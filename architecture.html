Core Philosophy

Claude Code follows a streaming-first architecture where API responses, tool execution, and UI updates flow continuously rather than in batch operations.

The system is designed around async generators enabling tools to execute as they stream in from the model, maximizing parallelism while maintaining safety guarantees.

Streaming Async Generators Real-time
Key Characteristics

  • Concurrency-aware tool execution
  • Multi-stage context compression
  • Hook-based extensibility

  • Dependency injection for testability
02 — Architecture

System Architecture

The architecture consists of four interconnected layers: Input/Output, Query Processing, Tool Execution, and External Services.

1 Input / Output Layer

Built with React + Ink for terminal UI rendering. Handles user input, message display, and real-time updates.

const { App } = require('./components/App.tsx')
const { REPL } = require('./screens/REPL.tsx')

Key components: Messages.tsx, StatusLine.tsx, Markdown.tsx

2 Query Processing Layer

Owns the query lifecycle via QueryEngine and the query() async generator. Handles compaction, streaming, and state management.

async query(messages, context) {
  yield* compact(messages)
  yield await callModel(messages)
  yield* runTools(streamingTools)
}
3 Tool Execution Layer

Self-contained tool modules with Zod schemas, permission models, and execution contexts. Tools can run concurrently if marked safe.

type Tool<Input, Output> = {
  name: string
  inputSchema: z.ZodType<Input>
  execute: (Input, ToolUseContext) => Promise<ToolResult>
}
4 External Services Layer

Integrates with Anthropic API, MCP servers, IDE bridges, and authentication providers (OAuth, API Key, AWS Bedrock, Azure Foundry).

Anthropic API MCP SDK OAuth 2.0 AWS Bedrock
03 — Tool System

Tool System Architecture

The tool system is the core execution mechanism, providing schema-validated, concurrency-aware, and hookable operations.

Tool Interface

Every tool is a self-contained module with input validation, permission control, and custom rendering.

type Tool<Input, Output> = {
  // Identity
  name: string
  description: string

  // Schema validation
  inputSchema: z.ZodType<Input>

  // Lifecycle
  isEnabled: () => boolean
  isConcurrencySafe?: (Input) => boolean
  execute: (Input, ToolUseContext) => Promise<ToolResult>

  // Optional rendering
  render?: (Output, Input) => React.ReactNode
}
Core Tools (45+)
  • AgentTool — Spawns sub-agents
  • BashTool — Shell command execution
  • FileReadTool — File system reads
  • FileEditTool — Code modifications
  • FileWriteTool — File creation
  • GlobTool — Pattern matching
  • GrepTool — Code search
  • WebFetchTool — HTTP requests
  • TaskCreateTool — Task management
Tool Categories
  • Core Tools — Always available
  • Conditional Tools — Behind feature flags
  • MCP Tools — From protocol servers
  • Plugin Tools — From extensions

Tools are sorted for "prompt cache stability" — consistent ordering for optimal token usage.

Concurrency Control

The StreamingToolExecutor partitions tools by concurrency safety:

// Concurrency-safe tools (read-only) run in parallel
const MAX_CONCURRENCY = 10

// Non-concurrent tools (writes) run serially
// with exclusive access to prevent conflicts

// Results are emitted in original order
results.bufferAndEmitInOrder()

04 — Query Engine

Query Engine & Processing

The query engine uses an async generator pattern for streaming-first processing with multi-stage compaction.

Query Loop Architecture

The query() function is an async generator that yields messages, stream events, and tombstones as processing happens.

submitMessage() User input enters QueryEngine
↓
processUserInput() Parse slash commands, model selection
↓
query() [async generator] Main processing loop
↓
callModel() Streaming API call
↓
StreamingToolExecutor Execute tools as they stream
↓
handleStopHooks() Post-execution hooks
State Management

State is carried between iterations for context continuity:

type State = {
  messages: Message[]
  toolUseContext: ToolUseContext
  autoCompactTracking: AutoCompactTrackingState
  hasAttemptedReactiveCompact: boolean
  pendingToolUseSummary: Promise<ToolUseSummaryMessage>
  stopHookActive: boolean
  turnCount: number
}
API Error Recovery

Multi-layer error recovery with circuit breakers:

  • Fallback Models — Retry with backup model
  • Streaming Fallback — Non-streaming retry
  • Token Budget — Track across compactions
  • Circuit Breaker — Stop after 3 consecutive failures
Dependency Injection

Query dependencies are injected for testability:

export type QueryDeps = {
  callModel: typeof queryModelWithStreaming
  microcompact: typeof microcompactMessages
  autocompact: typeof autoCompactIfNeeded
  uuid: () => string
}

export function productionDeps(): QueryDeps {
  return {
    callModel: queryModelWithStreaming,
    microcompact: microcompactMessages,
    autocompact: autoCompactIfNeeded,
    uuid: randomUUID,
  }
}
05 — Data Flow

Message & Tool Flow

How data moves through the system from user input to tool execution and back.

User Message Flow
User Input Command or message in REPL
↓
processUserInput() Parse commands, extract attachments
↓
Compaction Pipeline Snip → Microcompact → Autocompact → Context Collapse
↓
callModel() Streaming API request
↓
Tool Streaming Execute tools as they arrive
↓
Response Display to user
Tool Invocation Flow
// 1. ToolUseBlock arrives from model
ToolUseBlock { name: "Bash", input: { command: "ls" } }

// 2. Permission check
await canUseTool(tool, permissionContext)

// 3. Execute with hooks
await executePreToolHooks(tool)
result = await tool.execute(input, context)
await executePostToolHooks(tool, result)

// 4. Return to model for potential follow-up
return { result, content: "..." }
Context Management

Multiple compaction strategies maintain context within token limits:

  • Snip — Remove intermediate turns, preserve boundaries
  • Microcompact — Cache tool results by tool_use_id
  • Autocompact — Summarize when approaching limit
  • Context Collapse — Hierarchical summarization
Hook System

Extensible hooks at key points:

  • Pre-tool hooks — Before tool execution
  • Post-tool hooks — After tool execution
  • Permission hooks — Control tool access
  • Sampling hooks — After API response
  • Stop hooks — End of turn
06 — Design Patterns

Key Design Patterns

The architectural patterns that make Claude Code robust and maintainable.

Dead Code Elimination

Feature flags via Bun's bundle system enable conditional compilation.

const voiceModule = feature('VOICE_MODE')
  ? require('./voice/index.js')
  : null
bun:bundle Tree Shaking
Observable Store

Simple state management with immutable updates and subscriptions.

type Store<T> = {
  getState: () => T
  setState: (T => T) => void
  subscribe: (Listener) => () => Unsubscribe
}
Memoization

Expensive operations cached by session or cwd:

// Cached by working directory
const commands = memoize(
  () => loadCommands(),
  { key: process.cwd() }
)
Parallel Prefetch

Startup optimization by prefetching non-critical data in parallel.

await Promise.all([
  prefetchMDMSettings(),
  prefetchKeychain(),
  prefetchAPI(),
])
Multi-Agent Coordination

Coordinator pattern for orchestrating multiple workers.

// Workers report via XML notifications
<task-notification>
  <task-id>{agentId}</task-id>
  <status>completed</status>
  <result>{response}</result>
</task-notification>
Lazy Loading

Heavy modules loaded on-demand to reduce initial bundle.

// Dynamic import for heavy modules
const { OpenTelemetry } = await import(
  './services/telemetry.js'
)
